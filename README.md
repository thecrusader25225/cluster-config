## Kubernetes GitOps Configuration 

This repository contains Kubernetes manifests, ArgoCD applications, ingress configuration, and environment overlays for managing cluster infrastructure state.

This repository acts as the declarative infrastructure layer for the ML deployment platform.

The application platform responsible for dynamically generating and updating these manifests is available in the [ml-deployment repository](https://github.com/thecrusader25225/ml-deployment).

---

## Repository Responsibilities

This repository manages:

- Kubernetes manifests
- ArgoCD applications
- Kustomize overlays
- Environment-specific configurations
- Ingress routing
- Dynamically generated inference deployments
- GitOps reconciliation state

## Applications

### `apps/platform`

Contains Kubernetes manifests for the deployment platform itself.

Includes:

- Frontend deployments
- Backend deployments
- Platform services
- Horizontal Pod Autoscalers
- Environment overlays

---

### `apps/inference`

Contains dynamically generated inference workloads.

Each deployed model receives:

- Dedicated Deployment
- Dedicated Service
- Dedicated Kustomization resources

Example:

```text
apps/inference/dev/tinyllama-chat/
```

These manifests are automatically generated by the deployment platform.

---

### `apps/demo`

Contains sample/demo application manifests used for Kubernetes deployment testing and experimentation.

## Environment Configuration

### Base + Overlay Structure

The repository follows a Kustomize base/overlay architecture.

```text
base/
  ↓
overlays/dev
overlays/prod
```

This allows:

- Environment-specific configuration
- Independent image tags
- Namespace separation
- Deployment customization

---

## ArgoCD Structure

### App-of-Apps Pattern

ArgoCD applications are organized using an App-of-Apps architecture.

```text
argocd/
├── dev/
├── prod/
├── dev-root.yml
└── prod-root.yml
```

Root applications manage child applications for:

- Frontend
- Backend
- Inference workloads
- Ingress configuration

## GitOps Workflow

```text
Developer Push / Platform API
              ↓
     GitHub Repository Update
              ↓
           ArgoCD
              ↓
     Kubernetes Reconciliation
              ↓
       Cluster State Updated
```

Infrastructure changes are applied declaratively through Git commits rather than direct cluster mutation.

---

## Dynamic Inference Deployment Flow

```text
User uploads GGUF model
          ↓
ml-deployment platform generates manifests
          ↓
Generated manifests committed here
          ↓
ArgoCD detects repository changes
          ↓
Kubernetes reconciles desired state
          ↓
Inference workload deployed
```

---

## Networking

Ingress configuration is managed under:

```text
environments/base/
```

Routing structure:

```text
/api  → platform backend
/     → platform frontend
```

Ingress currently uses `sslip.io` host-based routing.

## Current Limitations

- External DNS currently relies on dynamic VM IPs
- ArgoCD currently uses polling instead of webhooks
- Generated inference workloads are environment-specific
- Persistent model storage is not yet configured
